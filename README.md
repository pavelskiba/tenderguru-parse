# Поиск тендеров (TenderGuru API) — автономный бот-мониторинг

Веб-приложение для поиска тендеров через API TenderGuru. Node.js + Express
отдаёт статический фронтенд (обычный HTML/CSS/JS) и проксирует запросы к
`https://www.tenderguru.ru/api2.3/export`, скрывая ключи API от клиента.

Фильтры (ключевые слова, регионы, разделы, закон 44-ФЗ/223-ФЗ/коммерческие,
тип заказчика, цена, площадка, ОКПД2) заполняются на странице **один раз** и
сохраняются на сервере. Дальше встроенный планировщик на `node-cron`
периодически (по заданному cron-расписанию) сам обращается к тем же фильтрам,
сравнивает найденные тендеры с уже отправленными ранее и рассылает только
новые — по Email, в Telegram и/или в Google Sheets. Планировщик работает
постоянно внутри Node-процесса, отдельный воркер/сервис не нужен.

## Как это работает

1. На странице настроек заполняете фильтры (как раньше), задаёте
   cron-расписание и отмечаете каналы доставки — жмёте **«Сохранить
   настройки»**. Всё сохраняется в JSON-файл на сервере (`lib/store.js`).
2. Кнопка **«Предпросмотр результатов»** — это старый ручной поиск: показывает
   таблицу тендеров по текущим значениям формы, ничего не сохраняет и не
   влияет на планировщик. Удобно, чтобы проверить фильтры перед сохранением.
3. Кнопка **«Проверить сейчас и отправить»** — запускает ту же логику, что и
   планировщик по расписанию, но немедленно. Удобно для проверки настроек
   Email/Telegram/Sheets, не дожидаясь ближайшего срабатывания.
4. По cron-расписанию (см. `lib/scheduler.js`) планировщик сам вызывает
   `/api/search` с сохранёнными фильтрами, сравнивает ID найденных тендеров со
   списком уже отправленных (`data/sent-ids.json`) и рассылает только новые.
   Каждый запуск пишется в журнал (`data/scheduler-log.json`), который виден
   внизу страницы настроек.

## Параметры API — статус проверки

Схема запроса подтверждена официальной документацией TenderGuru
(раздел «Список закупок (Россия)») и прямыми тестовыми запросами к API
с реальными ключами:

| Фильтр                     | Параметр TenderGuru                          | Статус |
|-----------------------------|-----------------------------------------------|--------|
| Ключевые слова               | `kwords`                                      | ✅ подтверждено |
| Формат ответа                | `dtype=json`                                  | ✅ подтверждено |
| Токен / refresh               | `api_code`, `refresh_key`                     | ✅ подтверждено |
| Закон (44/223/коммерческие)   | `f` = `44` / `223` / `kom`                    | ✅ подтверждено документацией |
| Регион                        | `r{ID}=1` (ID из справочника `config/regions.js`) | ✅ подтверждено (справочник получен через `mode=regions`) |
| Раздел                        | `c{ID}=1` (ID из справочника `config/sections.js`) | ✅ подтверждено (справочник `mode=cat`) |
| Площадка (ЭТП)                | `e{ID}=1` (ID из справочника `config/etp.js`)  | ✅ подтверждено (справочник `mode=eauc`, 100+ площадок, в проекте — отобранные крупные) |
| Цена от/до                    | `price1` / `price2`                           | ✅ подтверждено документацией |
| Дата размещения от/до         | `date_start` / `date_end`, формат `дд.мм.гггг` | ✅ подтверждено документацией |
| Пагинация                     | `page` (платная функция тарифа)               | ✅ подтверждено (размер страницы TenderGuru выбирает сам — не управляется отдельным параметром) |
| ОКПД2                         | `d1`/`d2`/`d3`/`d4` (сегменты кода через точку), только на `/api2.3/export/products` | ⚠️ подтверждено документацией, но не тестировано боевым запросом |
| Область поиска «в документации» | эндпоинт `/api2.3/export/documentation`     | ⚠️ подтверждено документацией, не тестировано боевым запросом |
| Область поиска «в продукции»    | эндпоинт `/api2.3/export/products`          | ⚠️ подтверждено документацией, не тестировано боевым запросом |
| Область поиска «в заголовке»    | отдельного режима в документации нет — используется обычный поиск | ⚠️ приближение |
| Тип заказчика «Государственный» | нет отдельного параметра — если закон не выбран явно, подставляется `f=44,223` | ⚠️ эвристика, не из документации |
| Несколько значений `f` одновременно | `f=44,223` (через запятую)              | ⚠️ предположение, не тестировано |

Если что-то из помеченного ⚠️ не сработает как ожидается — правки нужны
только в `config/params.js` и `lib/tenderguru.js` (функции `buildQuery`,
`resolveEndpoint`, `buildLawParam`), фронтенд трогать не придётся.

### Важное про ответ API

- Ответ — JSON-массив. Первый элемент вида `{"Total": "N"}` — общее число
  найденного по фильтру, не тендер. Остальные элементы — сами тендеры.
  Это уже обрабатывается в `lib/normalize.js`.
- На части тарифов поля `Customer`, `Etp`, `TenderLink` и `TenderNumOuter`
  приходят как текст-заглушка «Только для подписчиков портала: …» — это
  ограничение тарифа TenderGuru, а не ошибка приложения. Ссылка на источник
  поэтому берётся из `TenderLinkInner` (доступна на любом тарифе, ведёт на
  карточку тендера на tenderguru.ru).
- Реальные названия полей ответа: `TenderName`, `Customer`, `Category`,
  `Region`, `Price`, `EndTime`, `Fz`, `TenderLinkInner`, `TenderNumOuter`, `ID`.

## Стек

- Backend: Node.js + Express
- Frontend: статический HTML/CSS/JS в `public/` (без сборки)
- Один сервис, один `package.json` — годится для деплоя на Railway как
  единое приложение.

## Локальный запуск

```bash
npm install
cp .env.example .env
# впишите в .env свои TENDERGURU_API_KEY и TENDERGURU_REFRESH_KEY
npm start
```

Приложение слушает `process.env.PORT` (по умолчанию `3000`), откройте
http://localhost:3000.

## Переменные окружения

| Переменная                    | Назначение                                   |
|--------------------------------|-----------------------------------------------|
| `TENDERGURU_API_KEY`           | `api_code` — ключ доступа к API TenderGuru    |
| `TENDERGURU_REFRESH_KEY`       | `refresh_key` — код обновления/продления ключа |
| `PORT`                         | Порт сервера (на Railway задаётся автоматически) |
| `DATA_DIR`                     | Каталог для `settings.json` / `sent-ids.json` / `scheduler-log.json`. По умолчанию `./data`. На Railway укажите путь примонтированного Volume, иначе данные сбросятся при следующем деплое |
| `SCHEDULER_TZ`                 | Часовой пояс по умолчанию для cron-расписания (если не задан в настройках). По умолчанию `Europe/Moscow` |
| `RESEND_API_KEY`               | Ключ API [Resend](https://resend.com/api-keys) для отправки email (Railway блокирует исходящий SMTP на тарифах ниже Pro) |
| `RESEND_FROM`                  | Адрес отправителя. Без верификации своего домена в Resend доступен только `onboarding@resend.dev` |
| `EMAIL_TO`                     | Адрес(а) получателя писем с новыми тендерами  |
| `TELEGRAM_BOT_TOKEN`           | Токен Telegram-бота (получить у [@BotFather](https://t.me/BotFather)) |
| `TELEGRAM_CHAT_ID`             | ID чата/канала, куда бот отправляет сообщения |
| `GOOGLE_SERVICE_ACCOUNT_JSON`  | Содержимое JSON-ключа сервисного аккаунта Google (одной строкой), опционально — только если включён канал Google Sheets |
| `SPREADSHEET_ID`               | ID Google-таблицы (из её URL), опционально — только если включён канал Google Sheets |

Ключи **никогда** не хардкодятся в коде и не уходят на клиент — они
используются только на сервере (`lib/tenderguru.js`, `lib/delivery/*.js`) при
обращении к TenderGuru и каналам доставки.

### Настройка расписания

На странице настроек есть поле cron-выражения (стандартный 5-полевой формат
`минута час день месяц день-недели`) и список готовых пресетов. Например:

| Хочу                              | Cron              |
|-------------------------------------|--------------------|
| Каждый день в 11:00                 | `0 11 * * *`       |
| Каждые 6 часов                      | `0 */6 * * *`      |
| Каждый час                          | `0 * * * *`        |
| Каждый будний день в 09:00          | `0 9 * * 1-5`       |

Часовой пояс расписания фиксирован как `Europe/Moscow` (см. `SCHEDULER_TZ` для
изменения значения по умолчанию). Галочка «Включить автоматический запуск»
должна быть отмечена — без неё сохранённое расписание не активируется.

Планировщик за один запуск проверяет только первую страницу выдачи
TenderGuru по сохранённым фильтрам (как и обычный ручной поиск по умолчанию).
Если нужно отслеживать очень широкие фильтры с большим числом новых тендеров
за период между запусками — сузьте фильтры или увеличьте частоту расписания.

### Подключение Email

1. Зарегистрируйтесь на [resend.com](https://resend.com), создайте API-ключ и
   задайте `RESEND_API_KEY`, `RESEND_FROM` (по умолчанию `onboarding@resend.dev`,
   этого достаточно без верификации своего домена) и `EMAIL_TO` (куда слать).
2. На странице настроек отметьте канал «Email» и сохраните настройки.
3. Проверьте кнопкой «Проверить сейчас и отправить» — если новых тендеров нет,
   письмо не придёт (это ожидаемо, шлются только новые).

### Подключение Telegram

1. Создайте бота через [@BotFather](https://t.me/BotFather), получите токен →
   `TELEGRAM_BOT_TOKEN`.
2. Узнайте `chat_id` — для личного чата напишите боту и посмотрите
   `getUpdates`, для канала/группы добавьте бота туда и возьмите ID канала.
   Задайте `TELEGRAM_CHAT_ID`.
3. На странице настроек отметьте канал «Telegram» и сохраните настройки.

### Подключение Google Sheets (опционально)

1. Создайте сервисный аккаунт в Google Cloud Console, включите Google Sheets
   API, скачайте JSON-ключ.
2. Откройте нужную таблицу и предоставьте доступ на редактирование email
   сервисного аккаунта (`client_email` из JSON-ключа).
3. Задайте `GOOGLE_SERVICE_ACCOUNT_JSON` (содержимое JSON-ключа) и
   `SPREADSHEET_ID` (из URL таблицы).
4. На странице настроек отметьте канал «Google Sheets» и сохраните настройки.
   Новые тендеры дописываются новыми строками в первый лист таблицы.

## Деплой на Railway

1. Запушьте проект в GitHub-репозиторий (файл `.env` не попадёт в git —
   он в `.gitignore`; `.env.example` используется только как шаблон).
2. В Railway создайте новый проект → Deploy from GitHub repo.
3. Railway определит Node.js-проект (есть `package.json` со скриптом
   `start`, `Procfile` и `railway.json` для Nixpacks) и соберёт его
   автоматически.
4. В настройках проекта на Railway откройте **Variables** и добавьте нужные
   переменные из таблицы выше — как минимум `TENDERGURU_API_KEY` и
   `TENDERGURU_REFRESH_KEY`, а также переменные тех каналов доставки, которые
   собираетесь использовать (Email и/или Telegram обязательны для реального
   мониторинга, Google Sheets — опционально).

   Значения вписываются только в панели Railway — не в код и не в git.
5. Railway сам передаёт приложению переменную `PORT`; сервер уже слушает
   её (`server.js`).
6. **Персистентность данных.** Файловая система Railway эфемерна и
   сбрасывается при каждом новом деплое. Чтобы сохранённые настройки,
   расписание и список уже отправленных тендеров не терялись:
   - в Railway создайте **Volume** и примонтируйте его к сервису (например,
     на путь `/data`);
   - задайте переменную `DATA_DIR=/data`.

   Без этого шага после каждого редеплоя планировщик «забудет» уже
   отправленные тендеры и может продублировать рассылку один раз после
   деплоя, а сохранённые фильтры/расписание придётся вводить заново.
7. После деплоя откройте выданный Railway URL, заполните фильтры и
   расписание на странице настроек, нажмите «Сохранить настройки» — дальше
   планировщик работает в фоне сам, без необходимости держать страницу
   открытой.

## Структура проекта

```
server.js               # Express-сервер: /api/search, /api/dictionaries, /api/settings, /api/scheduler/*
lib/tenderguru.js        # Построение запроса к TenderGuru, выбор эндпоинта, ошибки
lib/normalize.js         # Приведение ответа TenderGuru к единому формату
lib/store.js             # Файловое хранилище настроек, отправленных ID и журнала запусков
lib/scheduler.js         # node-cron планировщик: дедупликация тендеров и рассылка новых
lib/delivery/email.js    # Отправка новых тендеров по email (Resend HTTP API)
lib/delivery/telegram.js # Отправка новых тендеров в Telegram (Bot API)
lib/delivery/sheets.js   # Запись новых тендеров в Google Sheets (сервисный аккаунт)
config/params.js         # Соответствие фильтров и параметров TenderGuru API
config/regions.js        # Справочник регионов (ID + название), mode=regions
config/sections.js       # Справочник разделов (ID + название), mode=cat
config/etp.js            # Отобранные крупные площадки (ID + название), mode=eauc
public/                  # Статический фронтенд (HTML/CSS/JS): фильтры, расписание, журнал запусков
data/                    # Рантайм-данные (settings.json, sent-ids.json, scheduler-log.json), не в git
```

## API

### `GET /api/dictionaries`
Возвращает списки регионов, разделов и ЭТП (каждый элемент — `{id, name}`)
для построения формы.

### `POST /api/search`
Тело запроса (все поля не обязательны, кроме постраничной навигации):

```json
{
  "kwords": "ремонт кровли",
  "kwordsWhere": "title",
  "okpd": "41.20",
  "laws": ["44fz", "223fz"],
  "customerType": "state",
  "etp": "16",
  "priceFrom": "100000",
  "priceTo": "5000000",
  "dateFrom": "01.01.2026",
  "dateTo": "31.12.2026",
  "regions": ["50", "77"],
  "sections": ["1"],
  "page": 1
}
```

`etp`, `regions`, `sections` — числовые ID из соответствующих справочников
(`config/etp.js`, `config/regions.js`, `config/sections.js`), а не названия.

Ответ:

```json
{
  "ok": true,
  "items": [
    {
      "title": "...",
      "number": "...",
      "customer": "...",
      "region": "...",
      "price": "...",
      "deadline": "...",
      "law": "...",
      "url": "..."
    }
  ],
  "page": 1,
  "onPage": 10,
  "total": 137,
  "hasMore": true
}
```

При ошибке: `{ "ok": false, "code": "...", "message": "..." }` —
обрабатываются неверный ключ, ошибки/лимиты API и пустая выдача.

### `GET /api/settings`
Возвращает сохранённые настройки мониторинга:

```json
{
  "ok": true,
  "settings": {
    "filters": { "kwords": "ремонт кровли", "regions": ["77"], "...": "..." },
    "schedule": { "cron": "0 11 * * *", "enabled": true, "timezone": "Europe/Moscow" },
    "delivery": { "email": true, "telegram": true, "sheets": false },
    "updatedAt": "2026-07-08T08:00:00.000Z"
  }
}
```

### `POST /api/settings`
Сохраняет фильтры, расписание и выбранные каналы доставки (тело — та же
структура, что в ответе `GET /api/settings`, без `updatedAt`). После
сохранения планировщик сразу перечитывает расписание — рестарт сервера не
нужен. Если `schedule.enabled: true`, а `schedule.cron` невалиден —
возвращается `400`.

### `GET /api/scheduler/status`
Возвращает последние записи журнала запусков планировщика (до 20, сначала
новые): время, тип запуска (`cron`/`manual`), статус, число найденных/новых
тендеров, каналы, куда отправлено, и текст ошибки, если была.

### `POST /api/scheduler/run-now`
Немедленно запускает ту же проверку, что выполняется по расписанию: поиск по
сохранённым фильтрам → сравнение с уже отправленными ID → рассылка новых по
включённым каналам → запись в журнал. Полезно для проверки настроек без
ожидания cron.

## Лимиты API

Тариф TenderGuru ограничивает число запросов в месяц (см. раздел
«Стоимость» в документации API — от 30 000/мес на тарифе «Стандарт» и
выше). Каждый поиск на фронтенде — один запрос к API; учитывайте это при
частом использовании и пагинации.

Каждое срабатывание планировщика — тоже один запрос к API. Например,
расписание «каждый час» даёт 24 запроса в сутки (~720/мес) на один набор
фильтров — сверьте частоту расписания с лимитом вашего тарифа.
